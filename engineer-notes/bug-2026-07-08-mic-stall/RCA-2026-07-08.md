# RCA: mic stall = generation-anchored PTS vs meeting-anchored file alignment

_Investigation 8 Jul 2026 (Claude), adversarially reviewed by GPT-5 (xhigh) - verdict: "causally sufficient, fits the timing unusually well". Supersedes the "Reading of the failure" section of REPORT.md; the observations there are all correct, but the two-defect theory (44.1 kHz format kill + writer never re-attached) is wrong. There is ONE root cause and no writer ever died._

## Root cause

**A 6 Jul regression (commit `cafca18`, the main-thread-livelock fix) changed the mic PTS anchor from meeting start to engine-generation start.**

- OLD: `pts = now - micStartTime`, set once at meeting start, survived engine rebuilds (nil'd only at stop).
- NEW: `MicAudioForwarder.deliver()` computes `pts = now - startedAt`, and `beginGeneration()` resets `startedAt` on **every watchdog rebuild** (`AppModel.attemptMeetingMicEngineStart` → `beginGeneration`).

The backend (`muesli_backend.py: write_aligned_audio`) aligns writes by PTS: `start_sample = pts_us * rate / 1e6`. A frame that lands **behind** `last_sample_index` has its overlapped front chopped; if the whole payload is behind, it is **silently dropped - no log line**.

So after every mid-meeting rebuild, mic PTS restarts at zero while the file position is at N seconds → the backend drops every mic frame until the new generation's clock catches up to N (needs N seconds of uninterrupted uptime). Any further rebuild resets the catch-up clock to zero. The Swift watchdog sees healthy tap frames (they ARE delivered and sent), so it declares recovery.

System audio is immune: its PTS is anchored once to the first SCStream buffer (CMTime delta) and the SCStream is never rebuilt. Same writer, same pipe, both healthy throughout - the drop happens inside the backend's alignment logic.

## Proof from backend.log (the live-tick interleaving)

Live ticks fire per 15 s of **new** audio per stream; the system ticks are an effective wall clock.

- Rebuild #1 (gen=5) at wall ~48.5 s: mic tick `45.15` at wall 45 → then **no mic tick while four system ticks pass** (wall 60/75/90/105) → mic `60.16` finally at wall ~109 → mic stays ~48.5 s behind system **for the rest of the recording**.
  - Prediction from the mechanism: resume at wall ≈ 48.5 + 45.15 ≈ 93.7, first new 15 s tick ≈ 109. Exact match.
  - Consequence: REPORT.md's "recovery worked" is wrong - rebuild #1 silently cost ~60 s of audio AND left everything after it written 48.5 s early in the file. **mic.wav is desynced from system.wav by ~48.5 s from 45 s onward**, which corrupts the final (reprocess) transcript's mic/system interleaving for the whole meeting, not just the lost tail.
- Rebuild #2 (gen=6) at ~59 min: catch-up would need ~59 min of uptime. Rebuild #3 (gen=7) 4 min later reset the clock again; the meeting ended 27 min after that → mic.wav frozen at 3534.08 s forever.
- Corroboration: mic.wav mtime 13:59 = last write at file position 3534.08 + 48.5 s desync = wall ~3582.6 s after the 12:59:17 start ≈ 13:59:00. ✓
- gen=7 came up at 16 kHz with healthy levels and still wrote nothing - which the format theory cannot explain and the PTS theory requires.

## The 44.1 kHz red herring

`engine.tap.format channels=4 sampleRate=44100.0` at gen=6 is NOT the killer: the meeting's **original** engine (gen=4, line 6 of the log) also negotiated 44.1 kHz and recorded fine for 59 minutes. `AudioConverterHelper.convertToInt16` downmixes to mono and resamples any rate to 16 kHz (vDSP linear interp). The rate flapping 44.1↔16 kHz across rebuilds is just the CoreAudio route churn that also triggered the stalls.

## Build verification

The deployed `/Applications/MuesliApp.app` binary (built 7 Jul 12:20) contains `MicAudioForwarder`/`RunLoopStormTripwire` (post-cafca18) but not `MicDeliverySurfaceDecision` (pre-c496e28) - i.e. the livelock-fix build with generation-anchored PTS. The backend runs from repo source (`.venv/bin/python -m diarise_transcribe.muesli_backend`), so the code read is the code that ran.

## Fix plan (GPT-5-reviewed priority order)

1. **(a) Meeting-anchored monotonic PTS** - forwarder gets a meeting-epoch set once at meeting start; `beginGeneration` keeps resetting per-generation liveness state but PTS = elapsed since meeting epoch. Use a **monotonic clock** (`ContinuousClock`/`mach_continuous_time`), not `Date` (NTP steps). PTS 0 = meeting-timeline zero (same timebase as system audio + screenshot events); a restart then produces a correct silence-padded gap. Compute PTS at capture time for the pending-buffer flush path (already the case). Add an end-to-end regression test simulating a generation restart at ~48 s.
2. **Kill the silent drop** - `write_aligned_audio` must log (rate-limited) every whole-frame drop and expose per-stream accepted/dropped counters in status lines.
3. **(d) Structured log lines** - ISO wall time + monotonic elapsed on every line; include stream, generation id, frame seq, raw PTS, accepted samples.
4. **(c) Sink-level watchdog** - drive it from backend **accepted-sample counters** (not `stat()` on the file); if tap frames flow but accepted samples don't advance for N s, surface a visible "mic not being recorded" UI warning.
5. **(b) Backend re-anchor recovery** - on sustained large PTS regression (epoch-reset signature), log loudly and realign to `max(last_sample_index, backend-receive-elapsed)` rather than dropping; GPT-5: treat as marked "timeline recovery", threshold well below 5 s (~250-500 ms alert, recover on sustained drops), and prefer shipping generation id + per-stream sequence numbers in the frame header so this isn't guesswork.

## GPT-5's residual caveat

The evidence is (very tightly) inferential because the drops are unlogged. For a courtroom-grade confirmation: tee/decode the framed stdin around a simulated rebuild and observe near-zero `pts_us` mic frames arriving while `last_sample_index` is far ahead, being whole-frame-dropped. The regression test in fix 1 doubles as this proof.

Alternatives considered and ruled out: frames not reaching Python (ruled out by tick resume at exactly R+L+15), stream mislabeling (system unaffected; mic resumed in-place), conversion failure (healthy levels + `tap.convert.fail` never logged), pipe/framing corruption (system continuous), backend writer death (file resumed growing after catch-up at gen=5; nothing restarted it).
